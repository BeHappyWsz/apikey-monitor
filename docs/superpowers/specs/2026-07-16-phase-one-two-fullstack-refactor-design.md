# 第一、二阶段前后端重构与可靠重启设计

## 1. 背景

当前项目是一个零第三方依赖的本地 API Key 管理面板，采用以下结构：

- `app.py`：HTTP 服务、路由、参数校验、后台检测调度和进程重启。
- `core.py`：粘贴解析、协议识别、连通性检测、模型级检测和配置导出。
- `db.py`：SQLite 数据访问与 `config.json` 配置管理。
- `monitor.py`：定时检测调度。
- `static/index.html`、`static/style.css`、`static/app.js`：原生前端。

项目近期增加了模型级检测、`check_model`、`config.json` 和监测线程退出等待逻辑。本次重构必须保留这些最新能力，并基于当前代码继续演进。

## 2. 已确认约束

1. 采用前后端同时调整的方案 B。
2. 保持 Python 标准库和原生 HTML/CSS/JavaScript，不引入第三方运行时依赖或前端构建链。
3. 列表接口暂时继续返回完整 API Key。
4. 暂时不处理 Git 仓库初始化和 Git 工作流。
5. 保留粘贴导入、协议识别、模型级检测、定时监测、编辑、批量操作和导出等现有功能。
6. 切换端口并确认重启时必须关闭旧端口服务，不能留下旧应用实例。
7. 新端口启动失败时自动回退并恢复旧地址和端口。

## 3. 目标

### 3.1 正确性与安全

- 修复 Anthropic 401/403 被判断为在线的问题。
- 双协议网关完成所有协议检测后再聚合总状态。
- 统一 Base URL 校验、规范化和接口路径拼接。
- 避免 Codex 导出产生 `/v1/v1`。
- 对导出值进行 Shell 安全转义。
- 编辑 Base URL 或 API Key 后清理旧检测结论。
- 限制请求体和第三方响应大小。
- 改善 SQLite 并发写入稳定性。

### 3.2 架构与维护性

- 从 `app.py` 拆出路由、业务服务和可靠重启职责。
- 从单体 `static/app.js` 拆出状态、API、列表、弹窗、设置等模块。
- 使用明确的任务状态描述批量检测过程。
- 保持模块边界简单，不引入框架式过度设计。

### 3.3 前端体验

- 修复筛选后全选误选隐藏记录的问题。
- 增加加载、检测中、批量任务进度和失败重试状态。
- 避免轮询竞态和不必要的全量重绘。
- 优化移动端卡片密度、工具栏、触控区域和编辑表单。
- 完善弹窗键盘操作与基本无障碍属性。

### 3.4 可靠重启

- 系统配置保存前校验目标端口。
- 重启过程中关闭后台监测和旧 HTTP Server。
- 确认旧端口释放后再启动新进程。
- 防止连续重启请求产生多个进程。
- 新服务启动失败时恢复旧配置，并在旧端口重新启动服务。
- 前端展示重启阶段并自动连接成功地址。

## 4. 后端设计

## 4.1 推荐目录结构

```text
apiKeyConfig/
├─ app.py                  # 启动参数、应用装配、HTTP Server 生命周期
├─ api/
│  ├─ __init__.py
│  ├─ handler.py           # BaseHTTPRequestHandler、统一响应和异常边界
│  ├─ router.py            # 精确路由匹配和分发
│  └─ validators.py        # JSON、ID、URL、设置参数校验
├─ services/
│  ├─ __init__.py
│  ├─ key_service.py       # Key CRUD、状态重置、单条/批量检测编排
│  ├─ task_service.py      # 检测任务、进度、同 Key 互斥
│  ├─ settings_service.py  # 设置保存和变更判断
│  └─ restart_service.py   # 可靠重启、端口释放确认和失败回退
├─ core.py                 # 解析、协议探测、模型探测、导出
├─ db.py                   # SQLite 和配置持久化
├─ monitor.py              # 定时任务触发，复用 task_service
└─ static/
```

拆分原则：

- `app.py` 不再承载具体业务路由。
- Handler 只负责 HTTP 协议处理。
- Service 负责业务规则和任务编排。
- `core.py` 保持无数据库依赖，便于单元测试。
- `monitor.py` 不自行复制检测逻辑，而是调用统一检测服务。

## 4.2 路由设计

路由使用精确匹配，避免仅通过 `startswith` 将错误路径识别成合法接口。

保留现有接口，并新增：

```text
GET  /api/tasks/{task_id}
GET  /api/system/restart/{restart_id}
```

调整批量检测响应：

```json
{
  "task_id": "check-...",
  "status": "queued",
  "total": 12,
  "completed": 0,
  "failed": 0
}
```

系统重启响应：

```json
{
  "restart_id": "restart-...",
  "status": "preparing",
  "old_url": "http://127.0.0.1:7878",
  "target_url": "http://127.0.0.1:8989"
}
```

所有 API 错误统一返回：

```json
{
  "error": "machine_readable_code",
  "message": "适合用户阅读的中文提示",
  "request_id": "..."
}
```

## 4.3 URL 规范化

新增统一函数：

```python
normalize_base_url(value) -> normalized_url
join_api_path(base_url, path) -> absolute_url
```

规则：

- 仅允许 `http` 和 `https`。
- 必须包含合法主机。
- 禁止换行、回车、空字符和其他控制字符。
- 去除末尾 `/`。
- 输入为 `/v1/models`、`/v1/messages` 等完整接口地址时，转换成协议根地址。
- 对已包含 `/v1` 的地址不重复添加 `/v1`。
- 保留合法的反向代理路径前缀。

探测与导出都调用同一套路径函数，避免规则分叉。

## 4.4 协议检测聚合

每个协议产生独立结果：

```python
{
  "protocol": "openai",
  "endpoint_exists": True,
  "authenticated": False,
  "status": "auth_error",
  "http_status": 401,
  "latency_ms": 120,
  "models": [],
  "error": "key rejected (401)"
}
```

聚合优先级：

1. 任意协议认证成功或明确可用：总状态 `up`。
2. 没有协议成功，但存在 401/403：总状态 `auth_error`。
3. 没有成功，但存在 429：总状态 `rate_limited` 或兼容映射为 `down` 并保存结构化错误；本次前端新增 `rate_limited` 展示。
4. 接口存在但处于 5xx 或部分功能失败：`degraded`。
5. 全部网络不可达、DNS 失败或超时：`down`。
6. 未检测：`unknown`。

Anthropic：

- 200：可用。
- 400：证明路由存在，但不能无条件证明 Key 可用；根据错误正文区分模型不支持与请求格式问题。
- 401/403：认证失败。
- 404：继续尝试下一个候选路径。
- 429：限流。
- 5xx：上游异常。

双协议必须完成所有必要检测后再聚合，不能在第一个协议认证失败时提前返回。

## 4.5 模型级检测兼容

保留当前：

- `check_model`
- `model_status`
- `model_latency_ms`
- `model_last_error`
- `model_last_check_at`

协议探测和模型检测使用相同的规范化 URL。模型检测失败不覆盖协议级在线状态，而是独立展示。

## 4.6 编辑后的状态处理

更新 Key 时先读取旧记录。若以下任一字段变化：

- `base_url`
- `api_key`

则同一事务内：

- `status = unknown`
- `supports_openai = 0`
- `supports_anthropic = 0`
- `models = []`
- `latency_ms = NULL`
- `last_check_at = NULL`
- `last_error = ''`
- 模型级检测结果重置为 `unknown`

接口支持：

```json
{
  "...": "...",
  "check_after_save": true
}
```

`check_after_save=true` 时，更新成功后创建检测任务；默认前端主按钮使用此模式。

## 4.7 检测任务服务

使用进程内任务注册表，不额外增加数据库表，以保持实现适中。任务包含：

- 任务 ID
- 类型
- 状态：`queued/running/completed/partial_failed/failed`
- 总数、完成数、失败数
- 创建和完成时间
- 简短错误列表

同一个 Key 同时只允许一个检测租约：

- 手动单条检测
- 批量检测
- 定时检测

发现已有任务时复用或跳过，防止重复向上游发送请求。

任务完成后保留有限时间，并定期清理，避免内存无限增长。

并发数统一读取设置中的 `concurrency`，不再硬编码 8。

## 4.8 SQLite 稳定性

连接统一设置：

```sql
PRAGMA foreign_keys=ON;
PRAGMA busy_timeout=5000;
```

WAL 在初始化阶段配置。数据库操作使用上下文管理器，确保异常回滚和连接关闭。

本次不引入完整迁移框架，但增加轻量 `PRAGMA user_version` 迁移入口，以便为最新模型字段和后续字段提供确定性升级。

## 4.9 请求大小限制

- 普通 JSON 请求最大 256KB。
- 粘贴导入最大 2MB。
- 批量候选和 ID 数量设置上限。
- 第三方响应限制读取字节数，模型列表仍最多保存 200 条。
- 超限返回 413 和明确错误信息。

## 4.10 导出

保留现有 Claude Code 和 Codex 格式，同时确保：

- URL 不重复拼接 `/v1`。
- Bash 值使用安全单引号转义。
- 非法 `fmt` 返回 400。
- 配置仍可在页面复制。

## 5. 可靠端口切换与重启设计

## 5.1 状态机

```text
idle
  -> validating
  -> shutting_down_old
  -> waiting_old_port_release
  -> starting_target
  -> verifying_target
  -> succeeded

任一步失败：
  -> restoring_old_config
  -> starting_fallback
  -> verifying_fallback
  -> rolled_back / failed
```

## 5.2 重启前校验

- 规范化目标 Host 和端口。
- 如果地址和端口没有变化，不执行进程重启，仅保存其他设置。
- 目标端口不是当前端口时，预检端口是否可绑定。
- 当前端口允许处于占用状态，因为占用者就是当前服务。
- 使用全局重启锁，已有重启任务时拒绝重复请求。

## 5.3 关闭旧服务

收到确认后：

1. 先响应重启任务 ID。
2. 停止接受新的敏感写操作。
3. 调用 `monitor.stop()` 并等待监测线程退出。
4. 调用当前 HTTP Server 的 `shutdown()`。
5. 主服务生命周期进入 `finally`，执行 `server_close()`。
6. 旧进程退出。

禁止仅启动新进程但保留旧 Server。

## 5.4 启动助手

可靠重启使用独立辅助进程，参数包含：

- 应用入口路径
- 工作目录
- 旧 Host/Port
- 新 Host/Port
- 旧配置快照
- 新配置快照
- 重启状态文件路径

辅助进程负责：

1. 等待旧进程 PID 消失。
2. 循环确认旧端口可绑定，确保已释放。
3. 启动目标配置的新进程。
4. 轮询目标 `/api/system/health`，确认启动成功。
5. 成功后将状态写为 `succeeded`。

## 5.5 自动回退

目标服务在限定时间内未健康启动时：

1. 终止目标子进程（若仍存活）。
2. 恢复旧配置快照。
3. 确认旧端口可用。
4. 使用旧 Host/Port 启动服务。
5. 轮询旧地址健康接口。
6. 成功时写入 `rolled_back`，包含目标启动错误。
7. 回退也失败时写入 `failed`，并保留可诊断日志。

配置持久化采用临时文件加原子替换，避免进程退出时留下半写状态。

## 5.6 防止应用残留

- 重启任务只允许一个。
- 辅助进程记录旧 PID 和目标 PID。
- 启动前确认旧 PID 已退出。
- 目标启动失败时明确终止目标 PID。
- 旧端口和目标端口分别进行绑定验证。
- 不通过扫描并终止系统中所有 Python 进程；只管理由本应用本次重启创建和记录的 PID。

## 5.7 前端重启体验

确认重启后显示不可关闭的进度弹窗：

```text
正在保存配置
正在关闭旧端口 7878
正在启动新端口 8989
正在验证新服务
```

前端依次尝试：

1. 目标地址健康接口。
2. 若目标未成功，尝试旧地址读取回退状态。
3. 目标成功则跳转新地址。
4. 回退成功则跳回旧地址，并提示“新端口启动失败，已恢复旧端口”。
5. 两边都不可达时展示手动启动命令和状态文件位置。

## 6. 前端模块化设计

## 6.1 文件结构

```text
static/
├─ index.html
├─ style.css
├─ app.js                 # 应用装配和启动
└─ js/
   ├─ api.js              # fetch、错误标准化、AbortController
   ├─ state.js            # KEYS、筛选、选中项、任务状态
   ├─ list.js             # 可见列表计算、卡片渲染和局部更新
   ├─ import.js           # 粘贴导入
   ├─ editor.js           # 编辑和保存后检测
   ├─ tasks.js            # 单条/批量检测任务轮询和进度
   ├─ settings.js         # 设置和可靠重启 UI
   ├─ dialogs.js          # 弹窗、焦点和 Esc
   └─ utils.js            # 转义、复制、时间、Toast
```

采用浏览器原生 ES Modules，不需要构建工具。

## 6.2 状态与刷新

状态层保存：

- 完整 Key 列表
- 当前搜索词与状态筛选
- 当前可见 Key ID
- 选中 ID
- 活跃任务
- 当前打开弹窗
- 最新加载序号

刷新策略：

- 首次加载显示骨架状态。
- 使用递增请求序号和 `AbortController`，旧响应不能覆盖新响应。
- 页面进入后台时暂停 5 秒轮询；恢复可见时立即刷新。
- 弹窗打开时允许更新后台数据，但不重建正在编辑的表单。
- 优先对状态、延迟和任务结果进行局部卡片更新；数据结构变化时才重新渲染整个列表。

## 6.3 筛选和选择

统一 `getVisibleKeys()`。全选仅作用于当前筛选和搜索结果。

显示：

```text
已选择 3 条（当前结果共 8 条）
```

切换筛选时不隐式删除已经选择的其他记录，但批量按钮旁明确显示总选择数；用户可一键“清除选择”。为避免误删除，删除确认框按状态列出可见与隐藏选择数量。

## 6.4 卡片与移动端

桌面端：

- 保留双列卡片。
- 默认信息结构更紧凑。
- 工具栏主动拆为操作行与筛选行，避免被动换行。

移动端：

- Hero 文案压缩，统计卡保留两列。
- 指标区采用两列。
- 默认展示名称、状态、URL、协议、延迟和最近检测。
- 模型列表、错误、备注和监测配置放入可展开详情。
- 卡片操作按钮触控区域至少 44px。
- 批量操作使用选中后出现的粘性底部操作栏。
- 编辑输入框与显示/复制按钮保持同一行，不拆成三行。

## 6.5 加载和任务反馈

- 初次加载：骨架卡片。
- 加载失败：页面内错误区和重试按钮。
- 单条检测：按钮 spinner、状态显示“检测中”，阻止重复点击。
- 批量检测：显示已完成/总数、失败数和进度条。
- 操作完成：保留 Toast，同时任务摘要可继续查看。
- 错误 Toast 显示更久，并支持展开详细信息。

## 6.6 弹窗和无障碍

- 弹窗增加 `role="dialog"`、`aria-modal="true"` 和标题关联。
- 打开时保存触发元素并聚焦首个输入控件。
- Tab/Shift+Tab 焦点限制在弹窗内。
- Esc 关闭非强制弹窗。
- 关闭后焦点恢复到触发元素。
- Toast 使用 `aria-live`。
- 所有交互增加 `:focus-visible` 样式。
- 重启进度弹窗在任务结束前不可通过遮罩或 Esc 关闭。

## 8. 实施顺序

### 第一步：建立测试保护

为当前检测和 URL 行为增加测试夹具，先复现已知错误。

### 第二步：后端核心与数据层

完成 URL、协议聚合、导出、编辑状态重置、SQLite 和请求限制。

### 第三步：服务层与路由

拆分 Handler、Router、Key Service、Task Service 和 Settings Service，并让定时监测复用任务服务。

### 第四步：可靠重启

实现重启状态机、旧端口释放、目标健康验证和失败回退。

### 第五步：前端模块化
