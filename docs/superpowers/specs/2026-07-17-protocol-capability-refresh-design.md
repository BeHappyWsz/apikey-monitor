# 协议能力检测与展示修正设计

**日期：** 2026-07-17  
**任务：** 修正密钥实例面板中协议能力的检测、持久化和展示规则。

## 背景

密钥实例面板通过 `supports_openai` 和 `supports_anthropic` 展示协议能力。当前协议探测会把部分非成功响应（尤其是 401/403）视为端点存在，因此某些对任意路径统一返回鉴权错误的中转服务会被误判为同时支持 OpenAI 和 Anthropic。后台健康监测还会保留历史能力标志，使错误结果无法随监测失败及时清除。

## 目标

1. 协议能力只代表最近一次探测中真正成功的协议。
2. 支持几种协议就展示几种，不设置优先级，也不限制数量。
3. 待检测或某协议探测失败后，不继续展示该协议的历史能力。
4. 手动检测和后台监测使用一致的能力更新语义。

## 非目标

- 不新增数据库字段或协议优先级字段。
- 不改变模型检测状态的含义。
- 不新增第三种协议。
- 不把鉴权失败、限流或服务异常解释为协议检测成功。

## 判定规则

每个已探测协议独立判定：

- 只有协议探测结果的 `status` 为 `up`，对应 `supports_*` 才为真。
- `auth_error`、`rate_limited`、`degraded`、`down` 和未知状态均不确认为协议能力。
- OpenAI 和 Anthropic 均成功时，两个能力字段都为真，前端展示两个标签。
- 仅一个协议成功时，只展示该协议。
- 没有协议成功时，两个能力字段都为假，前端显示“未确认”。

## 检测与刷新流程

### 手动检测

手动检测继续执行完整协议发现，探测所有已注册协议。本次结果覆盖数据库中的旧能力字段，因此已经失效的能力会被清除。

### 后台监测

后台监测也应让能力字段反映本次实际探测结果：

- 已探测协议成功则保留或设置对应能力。
- 已探测协议失败则清除对应能力，不保留历史成功标志。
- 为保证面板能够展示全部当前能力，后台监测应执行完整协议探测，而不是永久只探测历史上成功的协议。

后台监测仍不执行额外的模型调用，避免改变现有模型检测流量策略。

## 前端展示

卡片继续读取 `supports_openai` 和 `supports_anthropic`：

- 两者均真：显示 `OpenAI Anthropic`。
- 只有一个为真：显示对应协议。
- 两者均假：显示“未确认”。

前端不再根据整体连通状态自行推断协议；后端能力字段是唯一数据来源。

## 数据兼容性

沿用现有数据库字段，不需要迁移。已有错误能力数据会在下一次手动检测或后台监测完成后被新规则覆盖。

## 测试方案

后端单元测试至少覆盖：

1. OpenAI 成功、Anthropic 失败，只确认 OpenAI。
2. Anthropic 成功、OpenAI 失败，只确认 Anthropic。
3. 两个协议均成功，确认两种能力。
4. 401/403 不再确认协议能力。
5. 已有能力在后续监测失败后被清除。
6. 后台监测能够发现历史能力之外的新协议。

前端检查覆盖：

1. 单能力展示。
2. 双能力展示。
3. 无能力时显示“未确认”。
4. JavaScript 语法检查通过。

## 验收标准

- 面板展示的每种协议都在最近一次探测中成功。
- 待检测或全部探测失败时不展示任何协议标签。
- 支持多种协议时完整展示，不应用优先级。
- 手动检测和后台监测都能刷新并清除能力字段。
- 现有后端及前端相关测试通过。
